The Challenge

Building an SSL-Style Bus Compressor

The SSL G-Series bus compressor is an industry legend. It's been on more hit records than any other piece of outboard gear. Engineers spend years learning its behaviour.

We wanted that sound in Noise Engine ‚Äî so we studied the real thing and built our own implementation in SuperCollider, matching the documented specifications as closely as possible. Here's how.

"How accurate can software really be compared to ¬£3,000 hardware?"

The answer surprised us. The compression algorithm ‚Äî the actual gain reduction math ‚Äî is identical. What you're paying for in hardware is transformers, VCA character, and decades of engineering refinement. But the behaviour? That's just math. And math is free.

The Process

Five steps. Every time.

Step 01
Research the real thing

Not "what should a compressor do?" but "what does the SSL G-Comp actually do?" We dug into manuals, measured specs, and documented every fixed value: attack times (0.1, 0.3, 1, 3, 10, 30ms), release times (0.1, 0.3, 0.6, 1.2s, Auto), ratios (2:1, 4:1, 10:1).

No guessing. No "that sounds about right." Real specifications.

Step 02
Document before you code

MASTER_COMPRESSOR.md was written before a single line of DSP code. Signal flow diagrams. Parameter tables. Architecture decisions with rationale.

When you document first, you're forced to actually understand what you're building. The document becomes the spec. The code just implements it.

Step 03
Decide and document why

Every architectural choice is recorded. Why custom peak detection instead of SuperCollider's Compander? Because Compander doesn't do dominant stereo detection ‚Äî the key SSL behaviour where max(L,R) controls both channels.

Future you (or your collaborators) will thank you for writing down the "why."

Step 04
Single Source of Truth

Attack times defined once, in config. Not hardcoded in SuperCollider. Not duplicated in Python. One array, referenced everywhere.

When SSOT compliance hits 100%, there's nowhere for bugs to hide. Change a value in one place, it updates everywhere.

Step 05
Verify against spec

Does the GR meter move like an SSL? Does Auto release behave correctly ‚Äî fast for transients, slow for sustained compression? Does the sidechain HPF prevent bass pumping without affecting the audio path?

Test against what you documented, not against vibes.

Repeat

Same process: DJ Isolator EQ

The master section needed an EQ. Not a generic 3-band, but a proper DJ isolator ‚Äî the kind where you can completely kill a frequency band, not just turn it down.

Signal Flow
Input ‚Üí LR4 Split ‚Üí LO / MID / HI ‚Üí Gain + Kill ‚Üí Sum

Research: LR4 (Linkwitz-Riley 4th order) crossovers at 200Hz and 2kHz ‚Äî flat phase response when summed, no comb filtering.

Document: MASTER_EQ.md defines the crossover frequencies, gain ranges (¬±12dB), and kill switch behaviour before implementation.

SSOT: Crossover frequencies defined once in config, used by SuperCollider for the filters and Python for the UI labels.

Result: A proper DJ isolator. Kill the bass, the bass is gone ‚Äî not just quiet. Same process, same quality, different component.
